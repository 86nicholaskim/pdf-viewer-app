# PDF-to-HTML 변환 환경 비교: 브라우저 vs 서버

PDF를 읽어 HTML로 변환하는 작업은 실행 환경(Client-side vs Server-side)에 따라 큰 차이가 있습니다. 각 환경의 특징을 비교하여 최적의 선택을 돕습니다.

## 1. 비교 요약표

| 비교 항목 | 브라우저 (React Popup) | 서버 (Node.js/TS) |
| :--- | :--- | :--- |
| **주요 기술** | React Hook, Web Worker, `window.print()` | Node.js, `jsdom`, `node-canvas` |
| **실행 주체** | 사용자 개별 브라우저 | 중앙 서버 (Backend) |
| **인쇄 방식** | 브라우저 인쇄 대화상자 호출 | 서버에서 HTML 생성 후 전달/배치 작업 |
| **성능 부하** | 사용자 기기 리소스 사용 | 서버 CPU/Memory 사용 |
| **장점** | 구현이 매우 쉽고 즉각적인 반응 | 자동화 가능, 대량 처리, 보안 우수 |
| **단점** | 사양 낮은 기기에서 느릴 수 있음 | 서버 환경 설정(폰트 등)이 복잡함 |

## 2. 환경별 상세 특징

### 브라우저 환경 (Client-side)
*   **적합한 케이스**: 사용자가 직접 화면을 보면서 특정 페이지만 골라서 인쇄하거나, 실시간으로 내용을 수정(Replace)해서 바로 인쇄해야 할 때.
*   **데이터 흐름**: PDF 로드 -> 훅 처리 -> 가상 돔 렌더링 -> `window.print()`.
*   **특이사항**: CORS 정책에 민감하므로 외부 PDF URL을 불러올 때 서버 설정이 중요합니다.

### 서버 환경 (Server-side)
*   **적합한 케이스**: 수천 장의 PDF를 밤새 HTML로 변환해둬야 하거나, 사용자의 개입 없이 자동으로 보고서를 생성해서 메일로 보내야 할 때.
*   **데이터 흐름**: API 요청 -> 서버에서 PDF 파싱 -> HTML 문자열 생성 -> DB 저장 또는 파일 전송.
*   **특이사항**: 리눅스 서버 등에서 한글 폰트가 설치되어 있지 않으면 글자가 깨지는 현상이 발생하므로 시스템 폰트 설정이 필수입니다.

## 3. 결론 및 추천
*   **즉시성**이 중요하다면 **브라우저 팝업** 방식을 추천합니다. (현재 프로젝트 구조에서 가장 빠르게 구현 가능)
*   **배치 작업 및 자동화**가 중요하다면 **서버 측 변환** 방식을 도입해야 합니다.

인쇄가 목적인 경우, 브라우저에서 `media print` CSS를 잘 활용하면 서버에서 고생하며 레이아웃을 잡는 것보다 훨씬 쉽고 예쁜 결과물을 얻을 수 있습니다.
